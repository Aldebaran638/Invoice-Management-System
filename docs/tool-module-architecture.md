# 工具化模块架构说明

## 1. 核心概念

- 工具组：侧边栏中的分组节点，只负责导航分组，不直接承载业务数据。
- 工具：一个完整的业务能力，在侧边栏中有一个明确入口，在右侧展示区有对应内容页面。
- 路由：某个工具在前端中的访问入口。
- API 契约：某个工具的前后端明确接口约定。
- 模块：某个工具在后端中的实现边界。

## 2. 当前项目的基本决定

- 登录后，左侧永远是工具栏。
- 右侧永远是工具内容展示区。
- 当前前端视觉风格必须严格保留。
- 导航必须同时支持“直接工具”和“工具组”两种结构。
- 每新增一个工具，都必须前后端一起开发。

## 3. 前端映射规则

- 一个工具组可以包含多个工具。
- 一个工具对应 `frontend/src/routes/_layout` 下的一个路由。
- 工具导航统一在 `frontend/src/config/tool-navigation.tsx` 中注册。
- 通用视觉基础组件继续放在 `frontend/src/components/ui`。
- 工具自己的前端代码应放在 `frontend/src/tools/<tool-key>/`。

## 4. 后端映射规则

- 一个工具对应一个后端模块。
- 后端模块负责该工具的 schema、service、数据访问、router 和测试。
- 新工具的后端模块应创建在 `backend/app/modules/<tool-key>/`。
- 认证、配置、数据库会话等共享能力继续放在公共基础目录中。

## 5. 新增一个工具时的固定交付流程

### 第一步：架构设计

架构师 AI 必须先完成以下内容：

1. 定义工具 key、路由路径、侧边栏位置，以及它属于哪个工具组。
2. 定义该工具涉及哪些数据库实体，或者是否复用已有表。
3. 定义 OpenAPI 契约，包括：
   - 列表查询
   - 详情查询
   - 创建
   - 更新
   - 删除
   - 权限规则
4. 定义前端页面状态，包括：
   - 加载中
   - 空状态
   - 成功状态
   - 校验错误
   - 权限不足
5. 定义测试范围和禁止越界的边界。

架构师 AI 必须输出：

- API 契约简报
- 数据模型简报
- 路由与侧边栏注册简报
- 验收清单

### 第二步：后端 AI 实现

后端 AI 必须按下面顺序执行：

1. 阅读架构简报和工具 API 契约。
2. 确认是复用已有表，还是需要新增迁移。
3. 实现模块内的 schema、service、router。
4. 在后端 API 入口中注册 router。
5. 添加或更新测试，包括：
   - API 正常流程
   - 权限校验
   - 参数校验错误
   - 共享行为的回归测试
6. 重新生成或验证 OpenAPI 输出兼容性。
7. 输出 backend report，内容包括：
   - 修改了哪些文件
   - 暴露了哪些接口
   - 数据库是否变更
   - 风险点和边界情况

后端 AI 的边界规则：

- 不允许修改无关的前端文件。
- 不允许在未批准的情况下私自扩展 API 范围。
- 必须明确保留认证与权限规则。

### 第三步：前端 AI 实现

前端 AI 必须按下面顺序执行：

1. 阅读架构简报和已批准的 API 契约。
2. 在 `tool-navigation.tsx` 中注册工具或工具组。
3. 在 `frontend/src/routes/_layout` 下创建对应路由。
4. 构建该工具自己的 API 调用层、查询层和页面组件。
5. 严格保持当前视觉风格，包括：
   - 侧边栏语言一致
   - 布局节奏一致
   - 组件体系一致
   - 不允许风格漂移
6. 添加或更新 Playwright 测试，包括：
   - 路由可访问
   - 核心成功流程
   - 空状态或错误状态
7. 输出 frontend report，内容包括：
   - 修改了哪些文件
   - 新增了哪个路由
   - 使用了哪些 API
   - 风格安全说明

前端 AI 的边界规则：

- 不允许伪造契约中不存在的 API 字段。
- 不允许破坏全局设计语言。
- 不允许把新工具硬耦合进无关工具。

### 第四步：测试 AI 验证

测试 AI 每次都必须执行同一套检查流程：

1. 阅读架构师、后端、前端三份报告。
2. 检查代码修改是否超出批准范围。
3. 运行该工具相关的后端测试。
4. 运行该工具相关的前端测试。
5. 至少执行一条完整链路：登录 -> 侧边栏 -> 工具页面 -> 主操作。
6. 记录结果，包括：
   - 通过或失败
   - 是否越界
   - 是否有 UI 回归
   - 是否有契约不一致

如果发现任何越界问题，该任务必须退回给对应 AI 修复后再合并。

## 6. 仅新增工具组时的固定流程

如果当前需求只是新增一个工具组：

1. 架构师 AI 定义组名、图标、显示规则和子工具归属。
2. 前端 AI 只更新侧边栏注册和导航行为。
3. 后端 AI 默认不动，除非权限或路由规则也发生变化。
4. 测试 AI 验证侧边栏渲染、展开逻辑和路由可访问性。

## 7. 最低完成标准

一个新工具只有同时满足下面全部条件，才算真正完成：

1. 侧边栏入口出现在正确的工具组中。
2. 登录后路由可正常访问。
3. 前端页面状态完整。
4. 后端 API 通过校验和权限测试。
5. OpenAPI 客户端仍然可正常生成和使用。
6. 自动化测试全部通过。
7. frontend report 和 backend report 都已产出。
