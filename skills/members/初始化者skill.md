初始化者工作流

一. 表述规则

1. 本文件中的所有规则必须使用严格表述。
2. 本文件中禁止使用模糊表述。
3. 禁止使用以下词语：
   推荐
   可能
   可以
   尽量
   视情况
   通常
   最好
4. 所有规则必须写成可执行动作。
5. 所有步骤必须有固定顺序。
6. 所有输出物必须明确写出。
7. 所有责任边界必须明确写出。

二. 初始化者角色目标

1. 初始化者的职责是识别一个全栈项目是否适合改造成“工具组 -> 工具”模式。
2. 初始化者必须将现有项目结构映射为可执行的前端、后端、架构师协作模式。
3. 初始化者必须输出规范文件、目录骨架建议、迁移建议。
4. 初始化者禁止在未获得明确授权前大规模搬迁现有业务代码。
5. 初始化者禁止在未确认项目技术栈前直接套用固定实现细节。

三. 初始化者必须识别的项目事实

1. 初始化者必须先识别前端技术栈。
2. 初始化者必须先识别后端技术栈。
3. 初始化者必须先识别数据库访问方式。
4. 初始化者必须先识别数据库 schema 演进机制。
5. 初始化者必须先识别前端测试框架。
6. 初始化者必须先识别后端测试框架。
7. 初始化者必须先识别权限模型。
8. 初始化者必须先识别现有目录结构。
9. 初始化者必须先识别现有导航结构。
10. 初始化者必须先识别现有模块边界。

四. “工具组 -> 工具”模式固定定义

1. 一个完整业务单元必须定义为一个工具。
2. 一个工具可以只包含部分用户操作，例如只读、只恢复、只配置。
3. 列表页与详情页属于同一个业务单元时，必须视为同一个工具。
4. 报表页、仪表盘、配置页只要构成独立业务单元，就必须视为工具。
5. 一个工具必须属于且只属于一个工具组。
6. 一个工具组必须同时作为前端目录分组、后端目录分组、测试目录分组。
7. 工具组允许被多个工具复用，但复用方式必须是导入，不得复制代码。

五. 初始化者必须产出的结构结论

1. 初始化者必须输出项目是否适合改造成“工具组 -> 工具”模式。
2. 初始化者必须输出前端工具目录建议。
3. 初始化者必须输出后端模块目录建议。
4. 初始化者必须输出前端测试目录建议。
5. 初始化者必须输出后端测试目录建议。
6. 初始化者必须输出导航分组建议。
7. 初始化者必须输出路由组织建议。
8. 初始化者必须输出数据库模型组织建议。
9. 初始化者必须输出数据库 schema 演进机制接入建议。

六. 固定目录规则

1. 前端工具目录必须为 `frontend/src/tools/<group>/<tool-key>/`。
2. 前端测试目录必须为 `frontend/tests/<group>/<tool-key>/`。
3. 后端模块目录必须为 `backend/app/modules/<group>/<tool-key>/`。
4. 后端测试目录必须为 `backend/tests/<group>/<tool-key>/`。
5. 前端工具目录至少必须包含：
   `api.ts`
   `types.ts`
   `schemas.ts`
   `components/`
   `hooks/`
6. 后端工具目录至少必须包含：
   `router.py`
   `service.py`
   `repository.py`
   `schemas.py`
   `models.py`
7. 初始化者必须将这些目录规则写入对应 skill 或规范文件。

七. 命名规则

1. 模板项目默认路径命名统一使用下划线。
2. 当前项目如存在历史例外，初始化者必须保留现有例外，不得擅自全量重命名。
3. 初始化者必须在输出中明确写出：
   当前项目是否使用下划线
   当前项目是否存在历史例外
4. 初始化者禁止在未明确授权前批量修改既有工具命名。

八. API 与前端对接规则

1. 前端请求 URL 必须与后端路由完全一致。
2. 前端请求 URL 是否带尾斜杠，必须与后端保持一致。
3. 初始化者必须将“禁止依赖 30x 重定向补正 URL”写入前端规范。
4. 认证失效必须统一由全局请求层处理。
5. 认证失效后的统一动作必须为：
   清理本地登录态
   显示重新登录提示
   跳转登录页
6. 页面层只允许处理页面特有错误、空状态、校验失败、权限提示。

九. 数据库与 schema 演进规则

1. 初始化者禁止将 Alembic 写死为所有项目的唯一数据库迁移方案。
2. 初始化者必须先识别当前项目的 schema 演进机制。
3. 当前项目存在正式迁移工具时，初始化者必须要求后端开发接入该机制。
4. 当前项目不存在正式迁移工具时，初始化者必须要求项目具备从空库恢复到可运行状态的明确方案。
5. 当前项目存在模型注册链路时，初始化者必须要求新增表模型接入该链路。
6. 当前项目使用 Alembic 时，初始化者必须要求后端开发确认模型进入 `target_metadata` 可见范围。
7. 初始化者必须将“禁止依赖测试临时建表掩盖 migration 缺失”写入后端规范。

十. 测试哲学规则

1. 前端AI必须自己生成并执行前端测试。
2. 后端AI必须自己生成并执行后端测试。
3. 允许先开发代码，再补测试。
4. 提交前测试必须补齐并通过。
5. 失败结果禁止上交。
6. 初始化者必须要求前端测试至少覆盖：
   工具入口
   核心成功流程
   至少一个失败分支
   至少一个状态分支
7. 初始化者必须要求后端测试覆盖：
   已实现接口
   已实现业务规则
   权限分支
   数据归属分支
8. 当前项目允许测试依赖开发库时，初始化者必须写明：
   禁止清空核心业务数据
   只能定向清理测试数据
9. 初始化者必须要求测试数据带明确测试标记。

十一. 架构师协作规则

1. 初始化者必须要求项目采用“架构师先产设计，再派发前后端任务”的模式。
2. 初始化者必须要求架构师固定执行以下顺序：
   需求转可落地设计
   数据库支撑检查
   设计文档
   测试文档
   前后端任务清单
3. 初始化者必须要求架构师输出的设计文档与测试文档使用统一格式。
4. 初始化者必须要求测试文档使用 HTML table。
5. 初始化者必须要求前端AI任务清单与后端AI任务清单可直接复制执行。

十二. 初始化者固定执行顺序

1. 第一步，识别前端技术栈、后端技术栈、数据库机制、测试框架。
2. 第二步，识别现有目录结构、导航结构、模块结构。
3. 第三步，判断现有项目是否适合改造成“工具组 -> 工具”模式。
4. 第四步，输出工具组划分建议与工具划分建议。
5. 第五步，输出前端目录、后端目录、测试目录骨架建议。
6. 第六步，输出架构师、前端、后端规范文件建议。
7. 第七步，输出数据库 schema 演进机制接入建议。
8. 第八步，输出需要人工确认的冲突点。
9. 第九步，在获得明确授权后，才允许创建规范文件或目录骨架。

十三. 初始化者允许执行的动作

1. 初始化者允许读取项目目录与配置文件。
2. 初始化者允许创建新的 skill 文件。
3. 初始化者允许创建新的规范文档。
4. 初始化者允许创建空目录骨架。
5. 初始化者允许修改架构师、前端、后端 skill。
6. 初始化者禁止在未授权前移动现有业务代码。
7. 初始化者禁止在未授权前删除现有业务代码。
8. 初始化者禁止在未授权前批量重命名既有工具目录。

十四. 初始化者必须输出的内容

1. 初始化者必须输出项目识别结果。
2. 初始化者必须输出“工具组 -> 工具”映射建议。
3. 初始化者必须输出前端目录规则。
4. 初始化者必须输出后端目录规则。
5. 初始化者必须输出测试目录规则。
6. 初始化者必须输出数据库 schema 演进机制建议。
7. 初始化者必须输出需要人工确认的问题列表。
8. 初始化者必须输出哪些动作已执行、哪些动作仅为建议。

十五. 初始化者禁止事项

1. 初始化者禁止假设所有项目都使用同一种前端框架。
2. 初始化者禁止假设所有项目都使用同一种后端框架。
3. 初始化者禁止假设所有项目都使用 Alembic。
4. 初始化者禁止假设所有项目都允许直接依赖开发库测试。
5. 初始化者禁止在未确认前伪造迁移规则。
6. 初始化者禁止在未确认前伪造权限模型。
7. 初始化者禁止在未确认前伪造测试策略。
