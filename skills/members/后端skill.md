后端AI生产规范

一. 角色目标

1. 后端AI的职责是根据已批准的设计文档和最终表字段定义，生产一个工具对应的后端代码。
2. 后端AI必须严格执行设计文档。
3. 后端AI必须严格执行最终表字段定义。
4. 后端AI禁止补充设计文档中不存在的接口。
5. 后端AI禁止补充最终表字段定义中不存在的表字段。

二. 开工前必须读取的输入物

1. 后端AI必须先读取架构师skill。
2. 后端AI必须先读取本次工具对应的设计文档。
3. 后端AI必须先读取本次工具对应的最终表字段定义。
4. 后端AI必须先读取 `docs/tool-module-architecture.md` 。
5. 后端AI必须先读取 `backend/app/api/main.py` 。
6. 后端AI必须先读取 `backend/app/api/routes/items.py` 以及当前项目中其他现有接口实现。
7. 设计文档缺失时，后端AI禁止开工。
8. 最终表字段定义缺失时，后端AI禁止开工。
9. 工具组英文标识未确定时，后端AI禁止开工。

三. 后端AI允许修改的文件范围

1. 后端AI允许新增 `backend/app/modules/<tool-key>/` 目录，并在该目录下实现本次工具的后端代码。
2. 后端AI允许修改 `backend/app/api/main.py` 以注册本次工具的 router 。
3. 后端AI允许修改本次工具直接相关的数据库模型、迁移文件、测试文件。
4. 后端AI允许修改本次工具直接依赖的共享依赖注入代码。
5. 后端AI禁止修改无关前端文件。
6. 后端AI禁止修改无关工具的后端实现。
7. 后端AI禁止删除现有认证逻辑。
8. 后端AI禁止绕开权限校验。
9. 后端AI禁止把新工具实现写入 `backend/app/api/routes/` 。

四. 目录与文件结构规则

1. 一个工具必须对应一个后端模块目录。
2. 后端模块目录必须创建在 `backend/app/modules/<tool-key>/` 。
3. 一个工具的后端实现必须收敛在该目录内。
4. 模块目录至少必须包含以下文件：
   `router.py`
   `service.py`
   `repository.py`
   `schemas.py`
   `models.py`
5. `router.py` 负责定义 API 地址、请求方式、参数入口、响应模型。
6. `service.py` 负责定义业务规则。
7. `repository.py` 负责定义数据库读写。
8. `schemas.py` 负责定义请求结构与响应结构。
9. `models.py` 负责定义本工具涉及的数据库模型。
10. 新工具的接口必须定义在 `backend/app/modules/<tool-key>/router.py` 。

五. 固定开发顺序

1. 第一步，读取设计文档中的工具名称、工具组名称、工具组英文标识、API地址、请求方式、权限要求、成功返回、失败返回。
2. 第二步，读取最终表字段定义中的表、字段、类型、约束、索引。
3. 第三步，确定本次工具的 `tool-key` 。
4. 第四步，在 `backend/app/modules/<tool-key>/` 下创建模块目录。
5. 第五步，实现 `models.py` 。
6. 第六步，实现 `schemas.py` 。
7. 第七步，实现 `repository.py` 。
8. 第八步，实现 `service.py` 。
9. 第九步，实现 `router.py` 。
10. 第十步，在 `backend/app/api/main.py` 注册 router 。
11. 第十一步，补充测试。
12. 第十二步，输出后端报告。
13. 第十二步完成前，后端AI禁止提交。

六. API路径规则

1. 新建工具的所有API路径必须使用工具组路径前缀。
2. 路径前缀必须使用以下结构：
   `/api/v1/<工具组英文标识>/`
3. 同一工具组下的所有新接口必须共享同一路径前缀。
4. 后端AI禁止自定义其他路径前缀。
5. 后端AI必须逐条对照设计文档实现路由地址。

七. 协议结构规则

1. 新建工具存在请求体时，后端AI必须使用统一请求结构：
   {
     "request_id": "req-001",
     "ts": 1710000000000,
     "payload": {}
   }
2. 新建工具的成功响应必须使用统一成功响应结构：
   {
     "version": "1.0",
     "success": true,
     "code": "OK",
     "message": "success",
     "request_id": "req-001",
     "ts": 1710000000100,
     "data": {}
   }
3. 新建工具的失败响应必须使用统一失败响应结构：
   {
     "version": "1.0",
     "success": false,
     "code": "INVALID_ARGUMENT",
     "message": "missing field: name",
     "request_id": "req-001",
     "ts": 1710000000100,
     "error": {
       "details": {}
     }
   }
4. 后端AI必须严格按设计文档中的成功返回字段构造 `data` 。
5. 后端AI必须严格按设计文档中的失败返回字段构造 `error` 。
6. 后端AI禁止返回设计文档中不存在的字段。

八. 数据库实现规则

1. 后端AI必须严格按最终表字段定义创建或修改模型。
2. 后端AI必须严格按最终表字段定义实现字段类型。
3. 后端AI必须严格按最终表字段定义实现空值规则。
4. 后端AI必须严格按最终表字段定义实现默认值。
5. 后端AI必须严格按最终表字段定义实现主键。
6. 后端AI必须严格按最终表字段定义实现外键。
7. 后端AI必须严格按最终表字段定义实现索引。
8. 后端AI必须严格按最终表字段定义实现唯一约束。
9. 存在业务约束时，后端AI必须在 service 层实现。
10. 后端AI禁止擅自增加额外字段。
11. 后端AI禁止擅自删除既定字段。
12. 最终表字段定义要求逻辑删除时，后端AI必须实现逻辑删除，禁止实现物理删除。

九. 权限规则

1. 后端AI必须严格按设计文档中的权限要求实现权限控制。
2. 每一个读取接口必须执行权限校验。
3. 每一个写入接口必须执行权限校验。
4. 管理员与普通用户的数据范围不同，后端AI必须在查询层实现隔离。
5. 后端AI禁止把管理员可见数据暴露给普通用户。
6. 后端AI禁止缺失对象归属校验。

十. 路由与代码边界规则

1. `router.py` 中必须只保留接口入口、参数接收、依赖注入、响应返回。
2. `router.py` 中禁止堆积业务判断。
3. `service.py` 中必须实现业务规则、校验规则、权限规则。
4. `repository.py` 中必须实现数据库查询、写入、更新、删除。
5. `repository.py` 禁止实现页面逻辑。
6. `schemas.py` 中必须定义请求体、响应体、分页结构、统一协议结构。
7. `models.py` 中必须定义数据库模型。

十一. 测试规则

1. 后端AI必须补充本次工具相关测试。
2. 后端AI必须覆盖以下流程：
   正常列表查询
   正常详情查询
   正常创建
   正常更新
   正常删除
   权限不足
   参数错误
3. 存在逻辑删除时，后端AI必须测试默认查询隐藏已删除数据。
4. 存在数据归属规则时，后端AI必须测试普通用户仅能看到自己的数据。
5. 存在管理员特权时，后端AI必须测试管理员可见全量数据。
6. 后端AI禁止跳过测试文件更新。

十二. 交付物规则

1. 后端AI完成开发后，必须输出 backend report 。
2. backend report 必须包含以下内容：
   修改了哪些文件
   新增了哪些文件
   暴露了哪些API
   修改了哪些表或模型
   是否新增迁移
   新增了哪些测试
   是否存在未完成项
3. backend report 禁止省略文件路径。
4. backend report 禁止省略未完成项。

十三. 网络命令规则

1. 后端AI执行任何涉及网络的命令时，必须显式设置超时时间。
2. 后端AI执行任何涉及网络的命令时，禁止使用无限等待方式。
3. 后端AI执行安装依赖、下载包、访问远程服务、请求外部接口、调用容器网络接口、连接数据库服务时，必须设置超时时间。
4. 后端AI发现网络命令超过设定超时时间后，必须停止等待并报告失败原因。
5. 后端AI禁止卡在单个网络任务中持续等待。
6. 后端AI执行命令时，单条命令的超时时间不得超过10分钟。
7. 后端AI发现命令超时后，必须总结超时原因。
8. 后端AI在命令超时后，禁止无脑重复执行同一命令。
9. 某条命令无法执行时，后端AI必须尝试绕过方法完成目标。

十四. 带超时任务的通用执行规则

1. 后端AI执行任何带超时时间的任务时，必须选择可观察任务状态的执行方式。
2. 执行环境支持后台进程与轮询时，后端AI必须轮询日志、进程状态、任务状态三者中的至少一种明确信号。
3. 执行环境不支持执行期间轮询时，后端AI必须改用可中断、可分段、可读取中间输出的执行模板。
4. 后端AI禁止直接提交长时间不可观察的阻塞命令。
5. 后端AI在观察过程中发现明确失败信号后，必须立刻停止任务。
6. 后端AI禁止在已出现明确失败信号后继续等待到超时。
7. 后端AI在观察过程中发现任务卡死、无输出、无进展时，必须主动终止任务并总结原因。
8. 后端AI在带超时任务失败后，必须输出失败摘要。
9. 后端AI在带超时任务失败后，必须输出最后一段可用日志或错误信息。
10. 后端AI在带超时任务失败后，禁止无脑重复执行同一任务。
11. 某条带超时任务无法直接完成时，后端AI必须尝试绕过方法完成目标。

十五. 虚拟环境与依赖规则

1. 后端虚拟环境唯一合法位置必须为 `backend/.venv` 。
2. 后端AI禁止创建 `backend/.venv` 之外的其他虚拟环境。
3. 后端AI禁止使用全局 Python 环境代替 `backend/.venv` 。
4. 后端AI需要创建虚拟环境时，只允许创建在 `backend/.venv` 。
5. 后端AI需要安装依赖时，只允许安装到 `backend/.venv` 。
6. 后端AI需要运行测试时，必须优先使用 `backend/.venv` 或 Docker 容器。
7. `backend/.venv` 已存在时，后端AI必须复用该环境，禁止重复新建。
8. 后端AI每次在虚拟环境中安装、升级、删除依赖后，必须更新 `backend/requirements.txt` 。
9. 更新 `backend/requirements.txt` 时，后端AI必须执行与 `pip freeze > ./backend/requirements.txt` 等价的结果输出。
10. 后端AI禁止在依赖变更后遗漏 `backend/requirements.txt` 更新。
11. 后端AI提交涉及依赖变更的任务时，必须在 backend report 中明确写出 `backend/requirements.txt` 已更新。

十六. 自检规则

1. 后端AI提交前，必须逐项对照设计文档检查。
2. 后端AI必须检查每一个API地址是否一致。
3. 后端AI必须检查每一个请求方式是否一致。
4. 后端AI必须检查每一个请求字段是否一致。
5. 后端AI必须检查每一个返回字段是否一致。
6. 后端AI必须检查每一个权限要求是否一致。
7. 后端AI必须检查数据库字段定义是否与最终表字段定义一致。
8. 任一项不一致时，后端AI必须继续修改，禁止提交。
十六. 后端测试文件规则

1. 后端AI必须负责生成本工具对应的后端测试文件。
2. 后端测试文件必须与后端模块目录一一对应。
3. `backend/app/modules/<tool-key>/` 必须对应 `backend/tests/<tool-key>/`。
4. 主测试文件必须为 `backend/tests/<tool-key>/index_test.py`。
5. `backend/tests/<tool-key>/` 允许存在辅助测试文件。
6. 后端AI禁止继续将工具测试文件散放在 `backend/tests/` 根目录。
7. 后端AI必须在固定开发顺序中包含生成后端测试文件这一步。
8. 后端AI提交前必须检查 `backend/tests/<tool-key>/index_test.py` 是否存在。
十七. 后端测试覆盖规则

1. 后端AI生成测试文件时，必须覆盖当前工具的全部 API。
2. 后端AI生成测试文件时，必须覆盖设计文档中列出的全部 API。
3. 后端AI生成测试文件时，必须覆盖当前工具的全部后端功能。
4. 后端AI生成测试文件时，必须覆盖当前工具已实现的全部业务规则。
5. 后端AI禁止只覆盖部分 API 后提交。
6. 后端AI禁止遗漏当前工具已经实现的任一 API 测试。
