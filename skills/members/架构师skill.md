架构师工作流

一. 表述规则

1. 本文件中的所有规则必须使用严格表述。
2. 本文件中禁止使用模糊表述。
3. 禁止使用以下词语：
   推荐
   可能
   可以
   尽量
   视情况
   通常
   最好
4. 所有规则必须写成可执行动作。
5. 所有步骤必须有固定顺序。
6. 所有输出物必须明确写出。
7. 所有责任边界必须明确写出。

二. 架构师在工具开发中的固定前置流程

1. 架构师在开始设计工具之前，必须先将需求转换为可落地设计。
2. 架构师在完成可落地设计之前，禁止进入数据库检查阶段。
3. 架构师在最终表字段定义明确之前，禁止进入设计文档阶段。
4. 架构师在设计文档完成之前，禁止进入测试文档阶段。
5. 架构师在设计文档完成之前，禁止进入前端页面设计阶段。
6. 架构师在设计文档完成之前，禁止进入后端接口设计阶段。

三. 第一步：将需求转换为可落地设计

执行`skills\members\architect_skills\tool-intake.md` skill,得到skill输出的json内容.

四. 第二步：检查数据库是否支撑当前工具
执行`skills\members\architect_skills\tool-intake.md` skill,得到skill输出的json内容.


五. 第二步补充产物：最终表字段定义

1. 数据库支撑结论明确后，架构师必须产出最终表字段定义。
2. 最终表字段定义未完成前，禁止进入设计文档阶段。
3. 每一张新表都必须单独列出最终字段定义。
4. 最终表字段定义必须包含以下内容：
   表名称
   字段名称
   字段类型
   是否允许为空
   默认值
   主键
   外键
   索引
   唯一约束
   字段说明
5. 存在业务规则约束的字段，必须在最终表字段定义中明确写出约束。
6. 最终表字段定义必须与数据库支撑结论保持一致。

六. 第三步：创建设计文档

1. 设计文档属于第三步产物。
2. 第二步未完成前，禁止创建设计文档。
3. 最终表字段定义未完成前，禁止创建设计文档。
4. 设计文档必须建立在数据库支撑结论之上。
5. 设计文档必须建立在最终表字段定义之上。
6. 设计文档中的接口设计必须与数据库支撑结论保持一致。
7. 设计文档中的接口设计必须与最终表字段定义保持一致。
8. 设计文档完成后，方可进入后续前端、后端、测试设计阶段。

七. 设计文档版本与存放规则

1. 每次新增工具时，架构师必须先检查 `docs` 目录下当前最新版本号。
2. 架构师必须在输出中明确写出当前最新版本号。
3. 每次新增工具时，架构师必须为本次设计文档创建新的版本目录。
4. 新版本号必须等于当前最新版本号加一。
5. 设计文档必须存放在 `docs/v<新版本号>/` 路径下。
6. 设计文档文件必须存放在对应版本目录内，禁止直接放在 `docs/` 根目录下。

八. 设计文档规则

1. 设计文档必须记录接口设计。
2. 设计文档必须按一个接口一行编写。
3. 同一工具下存在多个接口时，工具名称列必须合并单元格。
4. 设计文档必须使用 HTML 表格。
5. 设计文档禁止使用 Markdown 原生表格。
6. 设计文档字段必须包括：
   工具名称
   API地址
   请求方式
   Path参数
   Query参数
   Body参数
   Header参数
   权限要求
   成功返回
   失败返回
   备注
7. 请求参数必须按以下四类分列记录：
   Path参数
   Query参数
   Body参数
   Header参数
8. 成功返回必须单独成列。
9. 失败返回必须单独成列。
10. 返回内容必须展开到字段级。
11. 成功返回中的字段必须逐项写出。
12. 失败返回中的字段必须逐项写出。
13. 备注列暂不限制写法。
14. 设计文档当前阶段禁止要求以下内容：
   请求样例
   成功响应样例
   失败响应样例

九. API路径规则

1. 新建工具的 API 路径必须与所属工具组相关。
2. 架构师必须在设计接口之前先定义工具组英文标识。
3. 工具组英文标识必须写入可落地设计说明。
4. 新建工具的 API 路径前缀必须使用以下结构：
   /api/v1/<工具组英文标识>/
5. 同一工具组下的所有新接口必须共享同一个工具组路径前缀。
6. 设计文档中的 API 地址必须使用工具组路径前缀。
7. 未定义工具组英文标识前，禁止输出 API 地址。

十. 新建工具的统一协议规则

1. 只要是新建工具，接口协议必须使用统一结构。
2. 新建工具中，所有带请求体的接口，其 Body 参数必须使用以下请求结构：
   {
     "request_id": "req-001",
     "ts": 1710000000000,
     "payload": {}
   }
3. 新建工具中，所有接口的成功响应必须使用以下成功响应结构：
   {
     "version": "1.0",
     "success": true,
     "code": "OK",
     "message": "success",
     "request_id": "req-001",
     "ts": 1710000000100,
     "data": {}
   }
4. 新建工具中，所有接口的失败响应必须使用以下失败响应结构：
   {
     "version": "1.0",
     "success": false,
     "code": "INVALID_ARGUMENT",
     "message": "missing field: name",
     "request_id": "req-001",
     "ts": 1710000000100,
     "error": {
       "details": {}
     }
   }
5. 旧工具不受本节规则约束。

十一. 测试文档规则

1. 测试文档属于设计文档之后的固定产物。
2. 设计文档未完成前，禁止创建测试文档。
3. 测试文档完成前，禁止进入前后端开发阶段。
4. 测试文档必须存放在对应版本目录 `docs/v<版本号>/` 。
5. 测试文档禁止直接放在 `docs/` 根目录。
6. 测试文档文件名必须为 `<工具名称>-测试文档.md` 。
7. 测试文档必须使用 HTML table 。
8. 测试文档禁止使用 Markdown 原生表格。
9. 测试文档必须按工具组织。
10. 同一工具下存在多个 API 时，`工具名称` 列必须合并单元格。
11. 同一 API 下存在多个请求方式时，`API地址` 列必须合并单元格。
12. 同一请求方式下存在多个测试样例时，`请求方式` 列必须合并单元格。
13. 测试文档字段必须包含：
   工具名称
   API地址
   请求方式
   测试样例
   预期结果
   备注
14. `测试样例` 列必须直接写完整 curl 。
15. `预期结果` 列必须写关键返回字段和值。
16. `备注` 列必须简要说明该测试样例的测试目的。
17. 架构师完成测试文档后，测试AI必须按测试文档逐条执行。
18. 测试AI禁止跳过测试文档中已列出的样例。
19. 测试AI执行完成后，必须根据测试文档给出通过与失败结果。

十二. 向前端AI与后端AI发派任务的固定格式

1. 架构师在设计文档完成后，必须同时生成前端AI任务清单与后端AI任务清单。
2. 架构师禁止只生成口头说明。
3. 架构师生成的任务清单必须达到可直接复制给AI执行的程度。
4. 架构师生成的任务清单必须使用固定结构。
5. 架构师生成任务清单时，必须分别输出前端AI任务清单与后端AI任务清单。
6. 前端AI任务清单必须包含以下部分：
   你是本项目的前端AI。你必须严格执行，不允许自由发挥。
   你必须先读取以下文件
   本次要开发的工具
   你的目标
   你允许修改的文件范围
   你禁止修改的内容
   固定开发顺序
   你必须遵守的实现规则
   完成后必须输出 frontend report
   提交前必须自检
   你的最终输出格式
7. 后端AI任务清单必须包含以下部分：
   你是本项目的后端AI。你必须严格执行，不允许自由发挥。
   你必须先读取以下文件
   本次要开发的工具
   本次数据库结论
   你的目标
   你允许修改的文件范围
   你禁止修改的内容
   固定开发顺序
   你必须遵守的API和协议规则
   本次工具必须实现的业务规则
   模块职责边界
   你必须补充的测试
   完成后必须输出 backend report
   提交前必须自检
   你的最终输出格式
8. 架构师在任务清单中必须明确写出工具组名称。
9. 架构师在任务清单中必须明确写出工具组英文标识。
10. 架构师在任务清单中必须明确写出工具名称。
11. 架构师在任务清单中必须明确写出设计文档路径。
12. 架构师在任务清单中必须明确写出相关 skill 路径。
13. 架构师在任务清单中必须明确写出必须读取的代码文件路径。
14. 架构师在任务清单中必须明确写出允许修改的文件范围。
15. 架构师在任务清单中必须明确写出禁止修改的内容。
16. 架构师在任务清单中必须明确写出固定开发顺序。
17. 架构师在任务清单中必须明确写出交付物名称。
18. 架构师在任务清单中必须明确写出自检项。
19. 架构师生成任务清单时，必须使用编号列表。
20. 架构师生成任务清单时，禁止省略文件路径。
21. 架构师生成任务清单时，禁止省略未完成项说明要求。
22. 架构师生成任务清单时，禁止使用模糊表述。
23. 架构师输出前端AI任务清单与后端AI任务清单时，实际输出格式必须与以下格式示例完全一致：
   前端AI任务清单:
   ```text
   前端AI任务
   ```
   后端AI任务清单:
   ```text
   后端AI任务
   ```
24. 架构师禁止使用与格式示例不一致的外层结构。
25. 前端AI任务清单标题必须精确写为 `前端AI任务清单:` 。
26. 后端AI任务清单标题必须精确写为 `后端AI任务清单:` 。
27. 前端AI任务清单内容必须包裹在 `text` 代码块中。
28. 后端AI任务清单内容必须包裹在 `text` 代码块中。
29. 架构师禁止省略 `前端AI任务清单:` 标题。
30. 架构师禁止省略 `后端AI任务清单:` 标题。
31. 架构师禁止省略 `text` 代码块。
十二. 前端测试文件与测试AI执行规则

1. 前端AI必须负责生成本工具对应的前端测试文件。
2. 前端AI生成的前端测试文件必须与工具目录一一对应。
3. 前端AI生成的前端测试目录必须为 `frontend/tests/<tool-key>/`。
4. 前端AI生成的主测试文件必须为 `frontend/tests/<tool-key>/index.spec.ts`。
5. 前端AI允许在 `frontend/tests/<tool-key>/` 下生成辅助测试文件。
6. 架构师向前端AI发派任务时，必须明确写出前端AI负责生成本工具对应的前端测试文件。
7. 测试AI必须执行前端AI生成的前端测试文件。
8. 测试AI执行前端测试时，必须执行 `frontend/tests/<tool-key>/index.spec.ts`。
9. 测试AI禁止跳过前端AI已生成的主测试文件。
10. 测试AI执行前端测试失败后，必须输出失败测试名称、失败原因、最后一段可用日志。

十三. 后端测试文件与测试AI执行规则

1. 后端AI必须负责生成本工具对应的后端测试文件。
2. 后端AI生成的后端测试文件必须与后端模块目录一一对应。
3. 后端AI生成的后端测试目录必须为 `backend/tests/<tool-key>/`。
4. 后端AI生成的主测试文件必须为 `backend/tests/<tool-key>/index_test.py`。
5. 后端AI允许在 `backend/tests/<tool-key>/` 下生成辅助测试文件。
6. 架构师向后端AI发派任务时，必须明确写出后端AI负责生成本工具对应的后端测试文件。
7. 测试AI必须执行后端AI生成的后端测试文件。
8. 测试AI执行后端测试时，必须执行 `backend/tests/<tool-key>/index_test.py`。
9. 测试AI禁止跳过后端AI已生成的主测试文件。
10. 测试AI执行后端测试失败后，必须输出失败测试名称、失败原因、最后一段可用日志。

十四. skill 文件修改权限规则

1. `skills/members/架构师skill.md` 只允许由架构师修改。
2. `skills/members/前端skill.md` 只允许由架构师修改。
3. `skills/members/后端skill.md` 只允许由架构师修改。
4. `skills/members/测试skill.md` 只允许由架构师修改。
5. 架构师向前端AI发派任务时，禁止要求前端AI修改任何 skill 文件。
6. 架构师向后端AI发派任务时，禁止要求后端AI修改任何 skill 文件。
7. 架构师向测试AI发派任务时，禁止要求测试AI修改任何 skill 文件。
8. 前端AI接收的任务必须是前端编码任务。
9. 后端AI接收的任务必须是后端编码任务。
10. 测试AI接收的任务必须是测试执行任务或测试代码任务。
