架构师工作流

一. 表述规则

1. 本文件中的所有规则必须使用严格表述。
2. 本文件中禁止使用模糊表述。
3. 禁止使用以下词语：
   推荐
   可能
   可以
   尽量
   视情况
   通常
   最好
4. 所有规则必须写成可执行动作。
5. 所有步骤必须有固定顺序。
6. 所有输出物必须明确写出。
7. 所有责任边界必须明确写出。

二. 架构师在工具开发中的固定前置流程

1. 架构师在开始设计工具之前，必须先将需求转换为可落地设计。
2. 架构师在完成可落地设计之前，禁止进入数据库检查阶段。
3. 架构师在数据库支撑结论明确之前，禁止进入设计文档阶段。
4. 架构师在设计文档完成之前，禁止进入测试文档阶段。
5. 架构师在设计文档完成之前，禁止进入前端页面设计阶段。
6. 架构师在设计文档完成之前，禁止进入后端接口设计阶段。

三. 第一步：将需求转换为可落地设计

1. 架构师必须先与开发者反复确认需求。
2. 需求存在任何歧义时，架构师必须继续追问，直到歧义消失。
3. 在需求未被完全澄清前，架构师禁止开始数据库、接口、前端、后端、测试设计。
4. 架构师必须逐项确认以下内容：
   工具解决的问题
   使用该工具的用户
   工具所属的工具组
   工具入口位置
   工具页面包含的功能
   每个功能的输入
   每个功能的输出
   权限边界
   本次必须实现的范围
   本次明确不实现的范围
   边界情况
5. 第一步完成后，架构师必须产出一份可落地设计说明。
6. 可落地设计说明必须包含以下内容：
   工具名称
   工具组名称
   页面入口
   页面功能列表
   用户角色
   权限规则
   输入输出说明
   本次实现范围
   本次不实现范围
   边界说明

四. 第二步：检查数据库是否支撑当前工具

1. 在可落地设计说明完成后，架构师必须检查当前数据库是否支撑该工具。
2. 架构师必须先检查现有表结构。
3. 架构师必须判断该工具属于以下哪一种情况：
   现有数据库可直接支撑
   现有数据库经过扩展后可支撑
   现有数据库无法支撑，必须新建表结构
4. 架构师必须逐项检查以下内容：
   是否存在可复用的表
   是否存在可复用的字段
   是否存在可复用的表关系
   是否缺少必要字段
   是否缺少必要关系
   是否缺少必要索引
   是否缺少完整业务表
5. 架构师必须明确该工具的数据流：
   谁创建数据
   谁查询数据
   谁修改数据
   谁删除数据
   谁拥有数据
6. 第二步完成后，架构师必须产出数据库支撑结论。
7. 数据库支撑结论必须包含以下内容：
   当前数据库是否支撑该工具
   支撑该工具所依赖的现有表
   需要新增的字段
   需要新增的关系
   需要新增的索引
   需要新增的表
   数据归属规则

五. 第三步：创建设计文档

1. 设计文档属于第三步产物。
2. 第二步未完成前，禁止创建设计文档。
3. 设计文档必须建立在数据库支撑结论之上。
4. 设计文档中的接口设计必须与数据库支撑结论保持一致。
5. 设计文档完成后，方可进入后续前端、后端、测试设计阶段。

六. 设计文档规则

1. 设计文档必须记录接口设计。
2. 设计文档必须按一个接口一行编写。
3. 同一工具下存在多个接口时，工具名称列必须合并单元格。
4. 设计文档必须使用 HTML 表格。
5. 设计文档禁止使用 Markdown 原生表格。
6. 设计文档字段必须包括：
   工具名称
   API地址
   请求方式
   Path参数
   Query参数
   Body参数
   Header参数
   权限要求
   成功返回
   失败返回
   备注
7. 请求参数必须按以下四类分列记录：
   Path参数
   Query参数
   Body参数
   Header参数
8. 成功返回必须单独成列。
9. 失败返回必须单独成列。
10. 返回内容必须展开到字段级。
11. 成功返回中的字段必须逐项写出。
12. 失败返回中的字段必须逐项写出。
13. 备注列暂不限制写法。
14. 设计文档当前阶段禁止要求以下内容：
   请求样例
   成功响应样例
   失败响应样例

七. 新建工具的统一协议规则

1. 只要是新建工具，接口协议必须使用统一结构。
2. 新建工具中，所有带请求体的接口，其 Body 参数必须使用以下请求结构：
   {
     "request_id": "req-001",
     "ts": 1710000000000,
     "payload": {}
   }
3. 新建工具中，所有接口的成功响应必须使用以下成功响应结构：
   {
     "version": "1.0",
     "success": true,
     "code": "OK",
     "message": "success",
     "request_id": "req-001",
     "ts": 1710000000100,
     "data": {}
   }
4. 新建工具中，所有接口的失败响应必须使用以下失败响应结构：
   {
     "version": "1.0",
     "success": false,
     "code": "INVALID_ARGUMENT",
     "message": "missing field: name",
     "request_id": "req-001",
     "ts": 1710000000100,
     "error": {
       "details": {}
     }
   }
5. 旧工具不受本节规则约束。
